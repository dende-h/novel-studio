# 05 — クラウド同期（optin・Freemium 有料軸）

> 2026-06 の grill セッションで方針を確定。`03-architecture.md §6`（バックアップ/同期）と `99-open-questions.md`（同期サーバ・収益化）を具体化するもの。**本書は MVP（第1版）の同期を定義し、E2E・同時編集マージは将来へ温存する。**

## 0. 位置づけ

- **optin**：既定は完全ローカル（サーバへ何も送らない）。ユーザーがアカウントを作り「同期する」と選んだときだけ有効。
- **Freemium の有料軸＝「クラウド束」**：無料＝ツール本体（書く＋出す＋ローカル保存）。有料＝**我々のサーバを使う機能の束**＝クラウド同期（アンカー）／複数端末／自動バックアップ／AI・MCP アクセス／（将来）容量・クラウド版履歴。原則は「サーバが要るもの＝有料、純ローカル＝無料」。アカウント＝課金接点。詳細は §1.2。
- ローカルファースト原則は不変：同期は IndexedDB 正本の**上に乗る複製**であって、置き換えではない。オフラインでも全機能が動く。

## 1. 確定事項

| 項目 | 決定 |
|---|---|
| 同期の形 | **双方向同期**（複数端末で行き来）。ただし**同時編集は想定しない**。 |
| 競合解決 | **単一セッションで同時編集を構造的に排除**するため通常は競合が起きない。残余（オフライン編集の取り残し）への保険として **Work 単位 LWW（`updatedAt`）＋負け版を local snapshot へ退避**のみ。話/ブロック単位マージ・CRDT は持たない。 |
| 暗号化 | **サーバ側 at-rest 暗号化・鍵はサーバ管理（＝パスワード忘れでも復旧可）**。E2E/ゼロ知識は将来の上級オプションへ温存。 |
| アカウント | **有料会員のみが持つ**（クラウド同期の identity ＋ 課金接点）。ゲストはアカウント不要・完全ローカルで全機能。 |
| セッション | **単一アクティブセッション（重複ログイン禁止・後勝ち）＋ログイン状態の永続化**。新端末でログインすると旧端末は自動ログアウト。ログアウトは①明示的ログアウト②別端末ログインの押し出しのみ → 同時編集を構造的に排除。 |
| 同期単位 | **Work 単位**（バンドル全体ではなく作品ごとに差分同期）。 |
| 同期トリガ | **2点のみ**：ログイン時＝全 Work の双方向同期／自動保存時＝編集中 Work の push（pull しない）。手動「今すぐ同期」は任意の保険。 |
| プラン境界 | **2段**：ゲスト（未登録・無料・ローカル無制限・同期なし）／会員（登録・有料・＋自動同期・複数端末・自動バックアップ）。詳細は §1.1。 |

## 1.1 アカウント・プラン境界と未ログイン時の挙動

アカウント＝**有料会員（＝同期する人）だけ**が持つ。ゲストはアカウント不要で、ローカルファースト哲学（書くのは無料・無制限）を一切損なわない。課金が守るのは「書く行為」ではなく「運ぶ・守る（同期／複数端末／自動バックアップ）」。

### プラン（2段）
- **ゲスト（未登録・無料）**：ローカルで **作品数無制限**に書く／出す（EPUB・なろう・カクヨム）／ローカル保存。同期なし。アカウント不要。
- **会員（登録・有料）**：上記すべて＋**自動同期／複数端末／自動バックアップ／AI・MCP アクセス**（クラウド束・§1.2）。ログイン＝同期の identity ＋ 課金接点。

> 「書くのは無制限・無料、運ぶ／守るのが有料」。1 作品ゲートのようなソフトゲートは置かない（書き始める前に離脱させない＝まず使われることを最優先）。

### ローカルデータ主権（最重要）
ログイン状態に関わらず、**ローカル（IndexedDB）にある作品は常に全件・閲覧/編集/書き出しできる**。アプリは未ログインを理由に既存データを隠さない・消さない。プラン境界がかかるのは **「同期」だけ**。

### 未ログイン時の挙動（会員がログアウト/セッション切れ/別端末ログインで押し出された場合）
- ローカルの作品は**全部そのまま見える・書ける・出せる**。
- 上部に「**同期停止中**」バナー＋「ログインして同期を再開」。会員はすべて課金者なので「課金してないのに停止」という誤解は起きない（無料の中間会員を持たない 2 段構成の利点）。ローカルに残した最終ログインのヒントで「○○として再ログイン」を促す（D-SYNC-SESSION で別端末ログインに蹴られた動線と共通）。
- **同期**：未ログインでは止まる。再ログインで pull → 再開。
- **新規作成・編集・書き出し**：ゲスト・会員とも制限なし。

### 利用者数の把握
会員＝課金者数は正確に分かるが、無料ゲストは匿名のため会員登録だけでは全体規模が見えない。全体を知るには匿名テレメトリが必要（§6 の未決・推奨方針あり）。

## 1.2 有料クラウド束（cloud bundle）

有料会員は単一プランのまま、**我々のサーバを使う機能をまとめて束**で提供する（段を増やさない）。原則＝**「サーバが要るもの＝有料、純ローカルで完結＝無料」**。

| 機能 | 位置づけ | 我々のコスト |
|---|---|---|
| **クラウド同期**（アンカー） | 端末間で Work を行き来（§3） | Workers＋D1＋R2 |
| **複数端末** | 同期の自然な帰結（単一アクティブセッション・§1） | 同上 |
| **自動バックアップ** | 同期済みデータがそのまま遠隔バックアップ | R2 |
| **AI・MCP アクセス** | 同期済み作品を**自分の AI**に読ませる（D-PLAN-AI） | 軽（保存テキストを返すだけ。**推論コストはユーザー持ち**） |
| 容量・クラウド版履歴（将来） | 同期容量上限の引き上げ・遠隔の版履歴 | R2 |

無料に残すもの（純ローカル＝ゲートしない）：書く・図鑑・ローカル保存・**ローカル版履歴（snapshot）**・**全書き出し（EPUB/なろう/カクヨム）**・**ファイル往復での AI 連携**（`01 §3.1`）。書き出しは MVP の核「出す」なので決してゲートしない。

### AI・MCP アクセス（read-only リモート MCP）
- **自前 AI は用意しない**（資金的制約）。AI 推論は全てユーザーの AI（ユーザーの課金）が行い、我々は**同期済み作品の保存テキストを返すだけ**。
- 形 = **リモート MCP（Streamable HTTP）**。同期インフラ（Workers＋R2＋auth＋サーバ鍵復号）上に**認証付き MCP エンドポイント**を生やす。Claude Desktop / claude.ai / Claude Code 等のリモート MCP に対応。
- UX = アプリが **MCP 設定（URL＋トークン）をコピー**できるボタンを出し、各自の AI クライアントに貼る。
- スコープ = **read-only**（`list_works` / `get_work` / `get_glossary`）。AI の編集結果は当面 import 往復で戻す（書き込み tool は将来）。
- **常駐プロセス不要**なので **Tauri を待たず同期サーバと同時期に出せる**（ローカル stdio MCP は Tauri 後段のまま）。
- 漏洩境界 = MCP で読ませた瞬間、本文はユーザーの AI プロバイダへ出る。**明示的 opt-in＋「あなたの AI に送られます」表示**を必須にする。無料ユーザーにはファイル往復（`01 §3.1`）を案内＝漏洩境界が自分で見える。

## 2. データモデル

### ローカル（IndexedDB・既存に追加）
- 正本は従来どおり Work JSON（`repo.saveWork`）。
- 各 Work の `updatedAt`（既存）を同期判定に流用。
- 端末ごとに `lastSyncedAt`（最後にサーバと突き合わせた時刻）を保持。

### サーバ（Cloudflare 前提）
- **D1**（メタ・同期判定の真実）：
  ```
  works(user_id, work_id, updated_at, deleted, blob_key, size, synced_at)
    PRIMARY KEY (user_id, work_id)
  ```
- **R2**（本体）：Work JSON バンドルを at-rest 暗号化して格納（`blob_key` で参照）。
- **サーバ**：Cloudflare Workers（Pages Functions）。認証トークンを検証し、Work 単位で出し入れ。

## 3. 同期プロトコル

1. **Push**：ローカルで `updatedAt > lastSyncedAt` の Work をアップロード（メタ＝D1、本体＝R2）。
2. **Pull**：サーバ側で `updated_at` がローカルより新しい Work をダウンロードし、`importWorks`（既存）で反映。
3. **競合（残余ケースのみ）**：単一セッション（§1）により同時編集は通常起きない。例外はオフライン編集中に別端末ログインされた取り残し（§3 末尾）。その場合は `updatedAt` の新しい方を採用し、**負けた方の Work をローカル snapshot（`snap:<workId>`・既存機構）へ退避**してから上書き＝丸ごと消失を防ぐ安全網。
4. 完了後に `lastSyncedAt` を更新。

### トリガ（D-SYNC-TRIGGER）
同期点は**2つだけ**。ユーザーは同期を意識しない:
- **ログイン時＝全体の双方向同期**：全 Work を突き合わせ、サーバが新しいものを pull、ローカルに未送信（`updatedAt > lastSyncedAt`）があれば push。
- **自動保存時＝編集中 Work の push**：既存の自動保存（`editorStore.save()`）にフックし、その作品だけをアップロード。**pull はしない**（編集中に他端末の変更を引っ張らない）。
- **オフライン**：push 失敗は無視してよい。未送信分は次のログイン/自動保存時に `updatedAt > lastSyncedAt` で自然に拾われる（専用の再送キューは持たない）。
- **手動**：「今すぐ同期」ボタンは保険として任意。
- ログアウト中は一切同期しない（optin 維持）。

> 単一アクティブセッション（§1）により、両端末が同時に編集状態になることはない。唯一の穴は**オフライン編集中に別端末でログイン**されたケース：オフライン端末は自動ログアウトを受け取れず、復帰時に push が弾かれ、再ログインの pull で未送信編集が上書きされうる。これを完全マージで防ぐのは過剰なので、§3-3 の snapshot 退避で軽減する、が本書の立場。

## 4. core 資産の再利用

既存の純ロジックがほぼそのまま土台になる（サーバ側の新規実装を最小化）:
- `exportBundle` / `importBundle`（Work 配列の直列化・検証）
- `importWorks`（id 単位の upsert＝非破壊マージ）
- snapshot 機構（負け版の退避先）

サーバ側に必要なのは「認証 ＋ Work 単位の blob 出し入れ ＋ `updated_at` 比較」に集約される。

## 5. 決定事項（確定済み）

| ID | 決定 |
|---|---|
| **D-SYNC-OPTIN** | 既定ローカル・optin。同期 OFF でも全機能がオフラインで動く。 |
| **D-SYNC-DIRECTION** | 双方向同期。ただし同時編集は非対応。 |
| **D-SYNC-SESSION** | 単一アクティブセッション（重複ログイン禁止・後勝ち）＋ログイン状態の永続化。新ログインで旧端末を自動ログアウト＝同時編集を構造的に排除。ログアウトは①明示的②別端末ログインの押し出しのみ。認証基盤にサーバ側セッション無効化を要求。 |
| **D-SYNC-CONFLICT** | 同時編集はセッションで排除済み。残余（オフライン編集の取り残し）のみ Work 単位 LWW（`updatedAt`）＋負け版を local snapshot へ退避。マージ/CRDT は持たない。 |
| **D-SYNC-CRYPTO** | サーバ側 at-rest 暗号化・鍵サーバ管理（復旧可）。E2E は将来。 |
| **D-SYNC-UNIT** | 同期単位は Work。バンドル全体送信はしない。 |
| **D-SYNC-STACK** | Cloudflare Workers ＋ D1（メタ）＋ R2（本体）。 |
| **D-SYNC-TRIGGER** | 同期点は2つ＝①ログイン時に全 Work 双方向同期 ②自動保存時に編集中 Work を push（pull しない）。未送信分は次回 `updatedAt > lastSyncedAt` で回収＝再送キュー不要。手動は任意の保険。 |
| **D-PLAN-TIERS** | 2段＝ゲスト（未登録・無料・ローカル無制限・同期なし）／会員（登録・有料・＋自動同期・複数端末・自動バックアップ）。会員登録＝有料。ソフトゲート（作品数ゲート）は置かない。 |
| **D-PLAN-LOCALDATA** | ローカルデータ主権。未ログインでも既存ローカル作品は全件閲覧/編集/書き出し可。プラン境界は「同期」にのみ適用。新規作成・編集・書き出し・ローカル保存は誰でも無制限・無料（隠さない・消さない）。 |
| **D-PLAN-BUNDLE** | 有料は単一プランで「クラウド束」を提供（段を増やさない）。束＝同期（アンカー）／複数端末／自動バックアップ／AI・MCP アクセス／（将来）容量・クラウド版履歴。原則「サーバが要るもの＝有料、純ローカル＝無料」。書き出しは無料に残す。詳細は §1.2。 |
| **D-PLAN-AI** | AI・MCP アクセスは有料クラウド束の一機能。形＝**read-only リモート MCP**（Streamable HTTP・同期インフラ上の認証付きエンドポイント・`list_works`/`get_work`/`get_glossary`）。**推論はユーザーの AI が行い我々は保存テキストを返すだけ**（自前 AI なし）。設定コピーで各自の AI に接続。常駐不要ゆえ Tauri を待たず同期と同時期に出せる（ローカル stdio MCP は Tauri 後段）。本文流出は opt-in＋明示表示。 |

## 6. 未決事項（次に詰める）

- [ ] **削除の伝播（tombstone）**：片端末の削除を pull 側へ伝える仕組み。D1 `deleted` フラグ＋保持期間。物理削除のタイミング。
- [ ] **認証方式**：magic link / OAuth / passkey / email+password。「サーバ最小」哲学から外部 Auth（Supabase/Clerk 等）か magic link を優先検討。**要件＝単一アクティブセッション管理（新ログインで旧セッション無効化）ができること**（D-SYNC-SESSION）。
- [ ] **価格と課金基盤**：プラン段は確定（D-PLAN-TIERS＝ゲスト無料／会員有料）。残るは価格・有料の同期容量上限・課金基盤（Stripe 等）。
- [ ] **利用者数テレメトリ**：無料ゲストが匿名のため、全体規模は会員数だけでは見えない。【推奨】最小・匿名・**原稿内容は送らない**・オプトアウト可能なテレメトリ（インストール/DAU・アプリ版・ロケール程度）を入れて「未公開原稿はローカルのみ」哲学と両立させる。要否と最小項目を確定する。
- [ ] **容量・レート制限**：1 ユーザーあたりの R2 容量上限、同期頻度の制限。
- [ ] **画像（data URL）込みの同期サイズ**：表紙/図鑑サムネは Work JSON 相乗り（方式A）なので同期 blob が膨らむ。圧縮/分離の要否。

## 7. スコープ外（将来・本書では実装しない）

- **E2E / ゼロ知識暗号化**（鍵をユーザーのみが持つ）＝将来の上級/有料オプション。
- **同時編集・リアルタイム共同編集**・話/ブロック単位の自動マージ（CRDT）＝単一アクティブセッションで構造的に排除（D-SYNC-SESSION）。
- **公開プラットフォーム側の同期**（読者向け・PF は別リポ・別設計）。
- **ローカル stdio MCP・MCP からの書き込み（編集 tool）**＝前者は Tauri 後段、後者は read-only の次段。本書の AI アクセスは **read-only リモート MCP** に限る（§1.2・D-PLAN-AI）。

## 8. 受け入れ条件（ドラフト）

- [ ] アカウント作成・ログインができ、ログアウト中は同期 UI が出ない（optin）。
- [ ] 端末Aで作った作品が、端末Bでログイン→pull すると復元される。
- [ ] 端末Bで編集→push→端末Aで pull すると最新が反映される（双方向）。
- [ ] 同期は自動で走る（手動操作なしに、ログイン時の全体同期と自動保存時の push で反映される）。
- [ ] 同じ作品を両端末で別々に編集すると、新しい方が採用され、**古い方が端末の履歴（snapshot）に退避**されている。
- [ ] 同期 OFF（未ログイン）でも執筆・書き出し・ローカル保存が全て動く。
- [ ] 未登録（ゲスト）でも作品数無制限に作って書ける・出せる（作品数ゲートなし）。
- [ ] 会員が未ログイン（ログアウト/セッション切れ/別端末ログインの押し出し）で開いても、ローカルの全作品が見え・編集/書き出しでき、「同期停止中」バナーとログイン導線が出る。
- [ ] ログイン状態は永続化され、再起動しても再ログインを求められない（ログアウトは明示的操作か別端末ログインの押し出しのみ）。
- [ ] サーバに保存されたデータが at-rest 暗号化されている。
